# Побудова повердінкових UML-діаграм для проєктування інформаційних систем

## Тема - Розробка системи контролю доступу до інформаційних ресурсів підрозділу

## 1. Діаграма варіантів використання (Use Case Diagram)

Ця діаграма описує межі системи та взаємодію між зовнішніми акторами та функціоналом системи.

* **Співробітник:** може авторизуватися, переглядати доступні ресурси та запитувати доступ до нових.
* **Адміністратор:** наслідує можливості користувача, але також має ексклюзивне право на керування доступами та перегляд журналів аудиту.
* **Include:** Авторизація є обов'язковою передумовою для подання запиту.

`code`

    useCaseDiagram
    actor "Співробітник" as User
    actor "Адміністратор" as Admin
    actor "Сервер Аутентифікації" as System

    package "Система контролю доступу" {
        usecase "Авторизація в системі" as UC1
        usecase "Запит на доступ до ресурсу" as UC2
        usecase "Перегляд дозволених ресурсів" as UC3
        usecase "Керування правами користувачів" as UC4
        usecase "Аудит логів доступу" as UC5
    }

    User --> UC1
    User --> UC2
    User --> UC3

    Admin --> UC4
    Admin --> UC5
    Admin --|> User

    UC1 -- System : "Перевірка токена"
    UC2 ..> UC1 : <<include>>

`code`

## 2. Діаграма послідовності (Sequence Diagram)

Демонструє динаміку взаємодії об'єктів під час виконання сценарію "Перевірка прав та отримання доступу".

Діаграма відображає логіку перевірки прав у реальному часі. Якщо БД повертає відсутність прав, система пропонує альтернативний шлях — створення запиту на доступ, що забезпечує гнучкість бізнес-процесу.

`code`

    sequenceDiagram
    participant User as Співробітник
    participant Client as Інтерфейс (UI)
    participant AC as Контролер Доступу
    participant DB as База Даних

    User->>Client: Запит на відкриття документа
    Client->>AC: validateAccess(UserID, ResourceID)
    activate AC
    AC->>DB: Отримати права (Permissions)
    activate DB
    DB-->>AC: Role: Read-Only
    deactivate DB

    alt Є права доступу
        AC-->>Client: Дозвіл (Access Granted)
        Client-->>User: Відображення ресурсу
    else Права відсутні
        AC-->>Client: Помилка (Access Denied)
        Client-->>User: Повідомлення: "Потрібен дозвіл"
        User->>Client: Натиснути "Запитати доступ"
        Client->>AC: createAccessRequest(UserID, ResourceID)
        AC-->>Client: Статус: Очікування
    end
    deactivate AC
    
`code`

## 3. Діаграма діяльності (Activity Diagram)

Описує алгоритм роботи системи під час обробки входу та перевірки безпеки.

Ця діаграма фокусується на логічних розгалуженнях. Вона чітко показує, що кожна дія (успішна чи ні) завершується логуванням, що є критично важливим для інформаційної безпеки підрозділу.

`code`

    activityDiagram
    start
    :Користувач вводить логін/пароль;
    if (Дані коректні?) then (так)
        :Генерація сесії;
        :Запит до інформаційного ресурсу;
        if (Рівень доступу достатній?) then (так)
            :Надати доступ до ресурсу;
            :Записати подію в лог (Success);
        else (ні)
            :Заблокувати доступ;
            :Запропонувати форму запиту;
            :Записати спробу в лог (Warning);
        end
    else (ні)
        :Помилка авторизації;
        :Спроб лишилося? (n-1);
    end
    stop
    
`code`
